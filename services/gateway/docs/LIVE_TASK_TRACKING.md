# Cortex Live Task Tracking Spec

For Atlas. Every detail is explicit. Do not fill gaps with assumptions — if something is unclear, ask.

---

## Overview

The filesystem mirrors task state. Folder names change to reflect completion. The dashboard reads the filesystem and the gateway DB together.

- DB = source of truth for state and metadata
- Filesystem = visual state via folder naming + content via README files
- Dashboard = reads both, renders the accordion

---

## Workspace Root

All project folders live under:

$CORTEX_HOME/projects/
This path is the workspace root. It is NOT configurable for now. Hardcode it or read it from the gateway config.

projects/
└── {project-name}/
    └── tasks/
        ├── phase-1/
        │   ├── PHASE-README.md
        │   ├── {task-uuid}/
        │   │   └── README.md
        │   └── {task-uuid}/
        │       └── README.md
        ├── phase-2/
        │   ├── PHASE-README.md
        │   └── {task-uuid}/
        │       └── README.md
        └── phase-{n}/
            ├── PHASE-README.md
            └── ...
### Naming Rules

- Project folder: whatever name the user gives it in the dashboard. Lowercase, hyphens for spaces. Example: mythric, cortex-v01, my-project
- Phase folder: phase-{n} where n is sequential starting from 1. Example: phase-1, phase-2, phase-3
- Task folder: UUID generated by the gateway at task creation time. Example: 9ac9a98b-1acc-4d52-b964-b29501c817ea
- Finished marker:  (finished) appended to folder name. Example: phase-1 (finished), 9ac9a98b-1acc-4d52-b964-b29501c817ea (finished)

## When Renaming Happens

Renaming is part of the state transition handler. Same function, same request. Not a separate process.

### On task_approve (in routes/cortex-tasks.js)

1. Update DB: status = 'approved', approved_at = now
2. Get task's project_id and phase_number from DB
3. Build paths:
   workspace = '$CORTEX_HOME/projects'
   project = getProject(project_id)
   taskDir = `${workspace}/${project.slug}/tasks/phase-${phase_number}/${task_id}`
   taskFinished = `${taskDir} (finished)`
4. If taskDir exists → rename to taskFinished
5. Check: count approved in phase === count total in phase?
   Yes →
     phaseDir = `${workspace}/${project.slug}/tasks/phase-${phase_number}`
     rename to `${phaseDir} (finished)`
6. Check: all phases finished?
   For each phase 1 to project.phase_count:
     count approved === count total?
   All yes →
     projectDir = `${workspace}/${project.slug}`
     rename to `${projectDir} (finished)`
7. Update task README with approval info
8. Update PHASE-README with task completion status
### On task_reject or task_reopen

1. Update DB: status = 'rejected' or 'pending'
2. Get task's project_id and phase_number
3. Build paths (same as above)
4. If `${taskDir} (finished)` exists → rename to `${taskDir}`
5. If `${phaseDir} (finished)` exists → rename to `${phaseDir}`
6. If `${projectDir} (finished)` exists → rename to `${projectDir}`
7. Update task README with rejection feedback
8. Update PHASE-README with task status change
Rejection always cascades UP. Unticking a task unticks its phase and project.

## Project Creation Flow

### What happens when a project is created in the dashboard

1. User enters project name and description in dashboard
2. Dashboard calls POST /api/projects with { name, description }
3. Gateway:
   a. Generates UUID for project ID
   b. Creates slug from name (lowercase, hyphens for spaces, strip special chars)
   c. Inserts into projects table with phase_count = 1
   d. Creates filesystem:

      mkdir -p $CORTEX_HOME/projects/{slug}/tasks/phase-1

   e. Creates PHASE-README.md for phase 1:

      # Phase 1

      ## Status: not started

      ## Tasks
      (none yet)

   f. Returns project object to dashboard

### What happens when a phase is added

1. User clicks "Add Phase" in dashboard
2. Dashboard calls POST /api/projects/:id/phases
3. Gateway:
   a. Increments phase_count on the project
   b. Creates filesystem:

      mkdir -p $CORTEX_HOME/projects/{slug}/tasks/phase-{n}

   c. Creates PHASE-README.md
   d. Returns updated project

## Task Creation Flow

### Every task MUST have a project and phase

Tasks cannot be created without project_id and phase_number. The POST /api/tasks and task_create MCP tool must require these fields.

{
  "title": "Fix auth hardcoded admin",
  "description": "Remove hardcoded admin user from ADMIN_USERS, use token-based role",
  "project_id": "uuid-of-mythric-project",
  "phase_number": 2
}

If project_id is missing → 400 Bad Request: project_id is required
If phase_number is missing → 400 Bad Request: phase_number is required
If project doesn't exist → 404 Not Found: project not found
If phase doesn't exist → 400 Bad Request: phase {n} does not exist for this project

### What happens when a task is created

1. Gateway generates UUID for task ID
2. Inserts into cortex_tasks with project_id and phase_number
3. Creates filesystem:

   mkdir -p $CORTEX_HOME/projects/{slug}/tasks/phase-{n}/{task-uuid}

4. Creates task README.md:

   # {title}

   ## Status: pending
   ## Assigned: unassigned
   ## Source: {human|agent}
   ## Created: {timestamp}

   ## Description
   {description}

5. Updates PHASE-README.md to include new task in the task list
6. Returns task object

## API Endpoints

### Projects

| Endpoint | Method | Auth | Description |
|----------|--------|------|-------------|
| /api/projects | GET | Token | List all projects |
| /api/projects | POST | Token | Create project (creates folder + phase-1) |
| /api/projects/:id | GET | Token | Get project detail |
| /api/projects/:id | DELETE | Admin | Delete project |
| /api/projects/:id/phases | POST | Token | Add new phase to project |
| /api/projects/:id/phases | GET | Token | List phases with completion status |

### Tasks (updated)

| Endpoint | Method | Body | Description |
|----------|--------|------|-------------|
| POST /api/tasks | POST | { title, description, project_id, phase_number } | Create task (required: project_id, phase_number) |
| POST /api/tasks/:id/approve | POST | { comment } | Approve + rename folder + cascade check |
| POST /api/tasks/:id/reject | POST | { reason, guidance } | Reject + remove finished markers + cascade |
| POST /api/tasks/:id/reopen | POST | { reason } | Reopen + remove finished markers + cascade |

All existing task endpoints stay the same. These are the ones that trigger folder renames.

## Dashboard Accordion Mapping

The dashboard reads the filesystem to build the accordion. Each level:

| Accordion Level | Source | Visual |
|----------------|--------|--------|
| Project card | projects/{name}/ or projects/{name} (finished)/ | Card with name. Green if finished. |
| Task Breakdown folder | projects/{name}/tasks/ | Single collapsible entry point |
| Phase row | phase-{n}/ or phase-{n} (finished)/ | Row with checkbox. Strikethrough if finished. Shows 3/8 tasks complete. |
| Task row | {uuid}/ or {uuid} (finished)/ | Row with status badge from DB. Checkbox + strikethrough if finished. |
| Task detail | {uuid}/README.md | Expand task to see full detail. Progress, files, stubs, review feedback. |

### Summary banner (top of project page)

Reads from DB:
- Project name
- Current phase: "Phase 2 of 5"
- Who's working on what: "atlas: task X, gerald: task Y"
- Completion: "12/24 tasks complete"
- Cost: total cost across all tasks

### Notes section

Reads from PHASE-README.md ## Notes section. Human can add notes via dashboard. Agents read them when they start work on a phase.

### Agent assignment banner

Reads from DB active tasks:
- "atlas is working on Phase 2, Task: Fix auth hardcoded admin"
- "zeus is reviewing Phase 2, Task: Fix auth hardcoded admin"
- "gerald is idle"

## Error Handling

- If rename fails (permission denied, path not found) → log error, DO NOT fail the DB update. The DB state is authoritative. A filesystem desync is fixable; a failed DB transaction is not.
- If task is created with a phase that doesn't have a folder → create the folder automatically
- If a project folder already has (finished) and a new task is created in it → remove the (finished) marker from the project and phase folders
- If the filesystem and DB disagree → the DB wins. Add a reconciliation endpoint POST /api/projects/:id/sync that rebuilds folder state from DB.

## What NOT to Build

- No automatic phase creation from task content — phases are created manually by the human
- No task reordering within a phase — they sort by created_at
- No drag-and-drop between phases — that's v0.2
- No automatic project creation — human creates projects in the dashboard
- No nested phases — phases are flat within a project
- No task dependencies — flat task list per phase for v0.1
