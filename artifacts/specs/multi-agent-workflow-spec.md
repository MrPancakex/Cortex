# Multi-Agent Tree Workflow Spec

## Background

In March 2026, Gerald drafted a v0.2 Operating Model for a 3-bot delegation system (Builder, Integrator, Verifier). Gemini provided a critique. Claude provided strategic analysis. This spec documents the agreed target architecture and the honest assessment of what needs to be built first.

## Gerald's v0.2 Operating Model (Source)

### The Triad
- **Builder (Bot A):** Writes implementation
- **Integrator (Bot B):** Checks wiring, ensures stubs are not present
- **Verifier (Bot C):** Runs verification gate, produces evidence pack, is the only one that can write DONE

### Evidence Pack (Anti-Hallucination Mechanism)
Physical artifacts required per task:
- `execution.log` — actual terminal output
- `state-proof.txt` — SQLite row IDs or file hashes proving mutations occurred

Verifier cannot self-report. Must point to specific artifacts.

### Status Protocol
Only the Verifier can mark DONE. Builder cannot mark its own work complete.

## Gemini's Critique (Key Points)

**Correct:**
- "Paper rules vs hard enforcement" — LLMs write strict rules but don't consistently follow them without physical guardrails
- "Grading your own homework" problem solved by Triad separation
- `check-no-stubs.sh` is a vulnerability if it uses grep-based detection (agent returns empty array to bypass)

**Missing:**
- How are handoffs actually orchestrated? Something must spin up Bot B with clean context after Bot A finishes
- Nothing stops Bot A from running `git commit` and bypassing the whole triad

## Claude's Strategic Analysis

**v0.2 is architecturally sound as a target state, but assumes infrastructure that doesn't exist yet.**

Gerald currently stalls on single sequential tasks. No proven orchestration layer can reliably spin up Agent 1, hand off to Agent 2, and trigger Agent 3 with clean context isolation and file ownership enforcement.

**Key reframe:** The Crucible (verify-task.sh) IS already the Verifier. It's root-locked, Gerald cannot touch it, and it does real HTTP validation. Don't rebuild that as "Bot C" — that's a downgrade from a hardened bash script to an LLM that can hallucinate pass verdicts.

**What's actually missing:** The Builder/Integrator split and the orchestration loop. Gerald currently is Builder AND Integrator AND (attempted) Verifier all in one context window. That's why stubs happen.

## Recommended Minimum Viable Architecture

**Two agents, not three:**

- **Agent 1 (Gerald):** Builder — writes implementation
- **Agent 2 (Auditor/Integrator):** Receives diff, checks wiring, runs The Crucible, produces evidence pack. Does not need to be creative — needs to be skeptical and mechanical. Good fit for a local model via Ollama (doesn't need Codex-level reasoning, needs checklist-following).
- **The Crucible:** Immutable hard gate underneath both. Neither agent can bypass it. Evidence pack generated from Crucible's output, not agent self-reporting.

## Enforcement Must Be Infrastructural

Key principle from both Gemini and Claude: behavioral compliance drifts when context compacts. Enforcement must be at the infrastructure layer:

1. Separate context windows for each agent (orchestrator spins up fresh context for Bot B)
2. File ownership: Bot A cannot write to Bot B's output directory
3. The Crucible as the immovable verification layer
4. Evidence pack generated by The Crucible's output, not self-reported by any agent

## Current Status / Prerequisites

Before implementing v0.2:
1. Hermes evaluation needed — can it reliably execute a 3-task sequence without stalling?
2. Orchestration primitive must be proven — something must manage context handoffs
3. The operating model should be shaped by what the orchestration layer can enforce, not the other way around

If Hermes fails: fallback is Claude Code + OpenClaw running dual agents — Gerald builds, Claude Code audits, Crucible gates.
